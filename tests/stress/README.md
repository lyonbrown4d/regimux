# RegiMux k6 stress tests

The stress suite runs against the Docker Compose integration environment and
uses the same real dependency artifacts as the integration tests, plus OCI
registry paths for manifest/blob cache behavior.

## Workload

- npm: `lodash@4.17.21`
- PyPI: `six==1.17.0`
- Maven: `commons-io:commons-io:2.16.1`
- Hot container target: `hub/library/busybox:1.36.1`
- Cold manifest target: `hub/library/alpine:3.19`
- Cold blob target: `hub/library/hello-world:latest`
- Multi-repo target set: `busybox`, `alpine`, and `hello-world`

The k6 script has fixed scenarios for:

- cold and hot container manifest requests
- cold and hot container blob range requests
- concurrent requests against the same blob digest
- mixed container requests across multiple repositories
- container `referrers` and `tags/list`
- hot npm, PyPI, Maven, and mixed-ecosystem traffic

## Commands

- `task stress:smoke`: short sqlite-backed script validation.
- `task stress`: default sqlite-backed load profile.
- `task stress:mysql`: load profile with MySQL metadata storage.
- `task stress:postgres`: load profile with Postgres metadata storage.
- `task stress:databases`: sqlite, MySQL, and Postgres load runs, then a comparison report.
- `task stress:compare`: regenerate comparison output from existing sqlite/mysql/postgres JSON reports.
- `task stress:run`: run the k6 script against an already-started integration environment.
- `task stress:test`: run deterministic parser, aggregation, and report-generation tests.

Reports are written to `tests/stress/reports` as Markdown and JSON. Generated
per-run reports are ignored by git because they are local-environment
measurements. The load comparison Markdown report is kept as a baseline
location and is regenerated by `task stress:compare`.

The sqlite/MySQL/Postgres load tasks sample the RegiMux Compose service with
`docker stats` and `docker inspect` while k6 runs. Gauge metrics report average,
nearest-rank p95, and peak. Network I/O, block I/O, and restart count report
reset-aware start/end deltas, and OOM status reports both observed and final
state.

## Profiles

- `smoke`: quick validation, low concurrency.
- `load`: default report profile.
- `stress`: higher concurrency profile for manual runs.

To run a custom profile against an already-started integration environment:

```sh
REGIMUX_STRESS_PROFILE=stress task stress:run
```

To change the requested byte range for blob scenarios:

```sh
REGIMUX_STRESS_BLOB_RANGE=bytes=0-262143 task stress
```

## Report Fields

Each per-run JSON report has:

- `schema_version`: report schema version.
- `generated_at`: report generation timestamp.
- `report_name`: output file stem.
- `profile`: `smoke`, `load`, or `stress`.
- `metadata_store`: `sqlite`, `mysql`, `postgres`, or `current`.
- `base_url`: RegiMux endpoint used by k6.
- `blob_range`: HTTP `Range` header used by blob scenarios.
- `artifacts`: fixed artifact and container target definitions.
- `overall`: total requests, request rate, failure rate, duration percentiles, and bytes received.
- `scenarios`: one row per k6 scenario with executor settings and metrics.
- `endpoints`: one row per endpoint tag with request, throughput, failure, and latency metrics.
- `warmup`: setup request duration metrics for hot-cache population.
- `resources`: RegiMux container CPU, memory, PIDs, network/block I/O, restart count, OOM status, sample count, and sampling interval.
- `thresholds`: k6 check summaries.
- `k6_summary`: raw k6 summary payload for deeper analysis.

Comparison JSON reports contain the source per-run reports, a union of scenario
rows, and a union of endpoint rows keyed by metadata store. The comparison
Markdown report renders overall totals, resource gauges/deltas, scenario p95/throughput, and endpoint
p95/throughput side by side for sqlite, MySQL, and Postgres.

Cold scenarios are short shared-iteration baselines. Treat their latency rows
separately from sustained hot-cache throughput rows when comparing stores.
